Akasha Backend Review

Scanner & Database — Performance, Stability, and Scalability Assessment
Review date: 2026-05-31  |  Scope: src/scanner.rs, src/db/, schema migrations, and supporting infrastructure
Mandate reminder: Nothing is off the table for redesign or refactoring. The user's collection is on the order of ~424,000 items across many subfolders. Akasha must be faster than a JSON-instead-of-DB approach by as wide a margin as feasible, and must avoid full database rebuilds wherever possible — especially once classifiers and semantic search are layered on top.

1. Executive Summary

Akasha's backend is architecturally sound for an MVP but contains several design decisions that will become painful — some severely — at the target scale. The developer has made genuinely good calls on WAL mode, async/UI separation, and per-subfolder resume logic. However, the scanner is effectively single-threaded for I/O, issues one database round-trip per file, and loads entire folder trees into memory without pagination. At 424k items, these choices compound into a scanner that could take hours instead of minutes, and a UI that stutters or OOMs when the user opens a large root folder.

The good news: most of these are local, fixable problems rather than fundamental architectural flaws. The database schema is clean, the self-referencing folder tree is the right abstraction, and the separation of concerns between db::media, db::folder, and scanner gives you room to operate.

2. What's Being Done Well

Good WAL Mode + Async/UI Separation

The pool is initialized with journal_mode = WAL and a 5-second busy timeout. This is exactly right: readers (the UI, thumbnail loader) are not blocked by the scanner's writes. Combined with channel-based communication back to the UI thread, the app remains responsive during scans.

Steelman: The developer correctly identified that a frozen UI is worse than a slow scan. WAL + async tasks is a disciplined, production-grade foundation for a single-developer project.

Good Per-Subfolder scan_complete Resume Logic

The scanner skips already-complete subdirectories via filter_entry, marks directories complete on exit, and performs per-folder orphan cleanup rather than root-wide deletion. This means a crash or cancellation mid-scan is recoverable without starting over.

Steelman: For a library measured in hundreds of thousands of files, a full rescan from scratch is unacceptable. The dir-stack tracking is a clever, lightweight way to get resume semantics without a complex job queue or persistent work log.

Good Relative Paths + Blake3 Hashing

Storing relative_path alongside absolute_path provides portability if root paths move. Blake3 is fast, verified, and the index on blake3_hash enables deduplication lookups. The UNIQUE(folder_id, relative_path) constraint is the correct natural key.

Steelman: These choices show the developer is thinking about data integrity and future features (e.g., "find duplicates") rather than just getting pixels on screen.

Good Additive Migration Strategy

Schema changes so far have been ALTER TABLE ... ADD COLUMN operations with sensible defaults. This is the correct long-term approach for a local database where the user cannot be expected to run manual migrations.

3. Critical Concerns

Critical The Scanner is Effectively Single-Threaded

Inside scan_folder, the for entry in walker loop processes one file at a time. For each file that needs updating, it:

  1. spawn_blocking to hash + decode dimensions
  2. await the result
  3. Issues a single UPSERT
  4. Moves to the next file

While the hashing runs on a blocking thread, the loop itself waits. There is no pipelining, no worker pool consuming a queue of files, and no parallel I/O. On a modern NVMe SSD with 16+ cores, you will be I/O-bound and CPU-idle for most of the scan.

Steelman: Sequential processing avoids complexity around ordering, backpressure, and partial failure. For an MVP with small test_imgs/ directories, it works fine. The developer may also have been cautious about SQLite write contention.
Impact: At 424k items, even a fast Blake3 hash (~1 GB/s) and image header read can easily take 5–15 ms per file. Serially, that's 35 minutes to 1.7 hours just for the mechanical work, before we even talk about the database. Parallelizing file I/O and hashing could drop this to under 5 minutes on fast storage.

Critical Per-File UPSERTs with No Write Batching

Every changed file executes a standalone INSERT ... ON CONFLICT ... DO UPDATE. SQLite WAL mode helps, but each statement is still a separate transaction boundary (implicit BEGIN ... COMMIT) unless wrapped in an explicit transaction. The code does not do this. At 424k files, that's 424k fsyncs or WAL checkpoints.

Steelman: Individual UPSERTs provide fine-grained error isolation: one bad file doesn't roll back a batch of 1,000. The yield_now() every 1,000 files does give the async runtime a chance to interleave other work.
Impact: SQLite on WAL can do ~1,000–10,000 transactions/sec on an NVMe drive, but implicit per-statement transactions are at the low end. 424k implicit transactions could take minutes to tens of minutes of pure DB overhead. Wrapping 1,000 UPSERTs in a single explicit transaction would speed this up by an order of magnitude.

Critical Entire Result Sets Loaded Into Memory

Both list_by_folder and list_by_folder_recursive call .fetch_all(), returning a Vec<MediaFile> that is then stored in app.media_items. The UI virtualizes rendering (only visible rows are drawn), but the Vec still holds every record. For 424k items at ~200 bytes each plus string allocations, that's well over 100 MB of heap before thumbnails or textures.

Similarly, the scanner bulk-loads existing media for the entire tree into a HashMap<String, (String, i64, Option<String>)>. For 424k entries, this HashMap alone could consume 200–400 MB.

Steelman: Keeping everything in memory simplifies the UI code dramatically — no pagination state, no cursors, no "load more" logic. For the MVP target of test directories with dozens of files, it's perfectly reasonable.
Impact: Opening a large root folder will cause a noticeable pause (fetching 400k rows), memory pressure, and potential OOM on lower-RAM systems. It also increases GC pressure from frequent allocations during the query.

High needs_update Logic is Brittle and Ignores modified_at

The current check is:

old_hash.is_empty() || *old_size as u64 != file_size || old_format.is_none()

This has three problems:

  1. No mtime comparison: modified_at is collected from disk but never used to skip unchanged files. It could short-circuit the vast majority of files.
  2. Same-size overwrite miss: If a file is modified in-place but happens to end up with the same byte count, it is silently ignored.
  3. Metadata repair trap: Because the skip is based on hash/size/format, adding a new column (e.g., EXIF orientation, color profile) requires a full rescan to populate it for existing files. The backlog notes this explicitly.
Steelman: Hash comparison is the most robust possible signal of content change. Mtime can lie (git checkouts, archive extractions, filesystem quirks). The developer prioritized correctness over speed, which is defensible for a data-hoarding audience who hates silent corruption.

High No Incremental / Live Updates

The notify and notify-debouncer-full crates are in Cargo.toml but completely unused. The only way to pick up new files is a full rescan (or restart if the folder is incomplete). Once a folder is marked scan_complete = true, it is invisible to the scanner until a manual rescan resets it.

Steelman: File watchers add complexity: debouncing, cross-platform path normalization, rename detection, and handling directories that appear/disappear. The developer deliberately deferred this to Phase 7, which is a sensible roadmap decision.

Medium Thumbnail Cache is Global-or-Nothing

PerFolder cache mode is defined but returns None (unimplemented). All thumbnails go into a single flat directory under ~/.cache/akasha/. At scale, this creates a directory with hundreds of thousands of files, which can stress filesystem metadata operations.

Medium delete_orphans_for_root is Dead, Buggy Code

The function exists in db/media.rs but is no longer called. The comment in SESSION_NOTES.md correctly notes it only matches direct children, not all descendants via CTE. Dead code with known bugs is technical debt.

4. Recommendations

Rec 1: Implement True Batch Transactions in the Scanner

Wrap chunks of UPSERTs in explicit BEGIN ... COMMIT transactions. SQLite can absorb thousands of rows per transaction with minimal overhead.

Pros
  • 10–50× faster writes on the same hardware
  • Minimal code change — add begin() / commit() around the loop
  • WAL mode handles concurrent readers gracefully during the transaction
Cons
  • A panic/crash during a batch loses the whole batch (acceptable for a resumable scan)
  • One bad row could theoretically fail the whole batch unless handled with savepoints

Rec 2: Parallelize File Discovery + Hashing with a Worker Pool

Replace the sequential loop with a bounded producer/consumer pipeline:

walkdir (producer) → bounded channel → N workers (hash + dimension) → bounded channel → DB inserter (batch commits)

The inserter task collects results and flushes a batch every N rows or every M milliseconds. Use tokio::sync::mpsc or crossbeam_channel depending on whether you want the workers to be spawn_blocking or plain OS threads.

Pros
  • Saturates both NVMe IOPS and CPU cores
  • Natural backpressure via bounded channels
  • Decouples I/O rate from DB commit rate
Cons
  • More moving parts: channel closures, backpressure handling, ordered vs unordered results
  • Order of DB inserts is no longer deterministic (usually fine)
  • More complex error propagation

Rec 3: Add Mtime-Based Fast Path + Schema Versioning Column

Change needs_update to:

  1. Compare modified_at and file_size first. If both match, skip.
  2. If they differ, hash and upsert.
  3. Add a schema_version INTEGER column to media_files. When a new metadata field is added, increment the app's schema version and re-scan files where schema_version < current without requiring a full hash.
Pros
  • Eliminates hashing for ~99% of unchanged files
  • Schema version allows targeted metadata backfills without full rescans
  • Still robust: size+mtime mismatch triggers full verification
Cons
  • Mtime can be unreliable on some filesystems or after restores
  • Requires storing modified_at as a first-class versioning signal, not just audit info

Rec 4: Implement Cursor-Based Pagination for Media Lists

Replace .fetch_all() with LIMIT / OFFSET or (better) cursor-based pagination keyed by rowid or (folder_id, relative_path). The UI already virtualizes rendering — it just needs the data layer to match.

Pros
  • Constant memory regardless of folder size
  • Faster initial load (first page appears instantly)
  • Enables infinite scroll or large-list survival
Cons
  • Complicates "jump to index" and random-access navigation (viewer prev/next across pages)
  • Thumbnail prefetching needs to work across page boundaries

Rec 5: Implement the notify File Watcher (Phase 7, but Sooner)

Integrate notify-debouncer-full to watch configured roots. On events:

  • Create/Modify: Process the single file through a fast path (no walkdir)
  • Delete/Rename: Remove or update the DB record directly
  • New directory: Mark parent incomplete and queue a targeted scan of just that subtree
Pros
  • Eliminates the need for periodic full rescans
  • Makes the app feel "alive" — new downloads appear automatically
  • Dramatically reduces wear on large spinning-rust archives
Cons
  • Platform-specific edge cases (network mounts, FUSE, bind mounts)
  • Needs careful debouncing and coalescing
  • Watcher descriptors can be resource-intensive at extreme scale

Rec 6: Shard the Thumbnail Cache

Instead of {hash}_{size}.webp in one flat directory, use a two-level prefix: {hash[0..2]}/{hash[2..4]}/{hash}_{size}.webp. This keeps any single directory under ~65k entries. Also implement the PerFolder cache mode for users who keep media on external drives and want the cache colocated.

Pros
  • Prevents filesystem metadata bottlenecks
  • Trivial to implement; no schema changes
Cons
  • Harder to manually browse the cache directory (who does this?)

Rec 7: Introduce a scan_queue Table for Robust Resume

Replace the implicit dir-stack resume with an explicit work queue:

CREATE TABLE scan_queue (
    id INTEGER PRIMARY KEY,
    folder_id INTEGER NOT NULL REFERENCES folders(id),
    status TEXT NOT NULL CHECK(status IN ('pending','running','complete','failed')),
    started_at DATETIME,
    completed_at DATETIME
);

On startup, populate pending entries for any folder where scan_complete = false. A background task drains the queue. This makes resume crash-proof and enables parallel scanning of unrelated roots.

Pros
  • Crash at any point → restart picks up exactly where it left off
  • Enables parallel scanning of multiple roots
  • Visible in UI: "3 of 47 folders remaining"
  • Foundation for future background jobs (classifier inference, transcoding)
Cons
  • More schema and state to manage
  • Overkill if the app remains single-user/single-process forever

Missing Indexes

Column Type Notes

file_size and dimensions are stored as INTEGER (maps to i64 in sqlx). This is fine — SQLite has no unsigned type. The conversion noise at the Rust boundary is minor and already handled. However, consider adding CHECK(file_size >= 0) if you want to catch bugs early.

FTS5 and Vector Tables (Future)

The roadmap mentions FTS5 for text search and either sqlite-vec or in-memory HNSW for vector search. If you stay with SQLite (recommended for single-file portability), adding these as virtual tables in a migration is straightforward. Do it before the first 424k-item scan so you don't have to backfill.

6. Strategic Remarks

On "Faster Than JSON"

Currently, Akasha's DB is not yet faster than a well-designed JSON/metadata approach for the narrow case of "read all files in a folder." A flat JSONL file with an in-memory HashMap would beat SQLite on raw read throughput for small-to-medium folders because it avoids query parsing and row deserialization. Where SQLite wins — and where Akasha must lean into it — is:

  • Indexed lookups (find by hash, find duplicates)
  • Complex filtering (recursive CTEs, tag intersections, vector similarity)
  • Partial reads (pagination, not loading the whole world)
  • Concurrent access (thumbnail cache hits during scan writes)

To justify the SQLite dependency, you need to start using these capabilities. Add pagination. Add fast-path mtime queries. Plan for FTS5. Otherwise you are carrying database overhead without getting database benefits.

On Single-Developer Nimbleness

Most of the recommendations above (batch transactions, mtime fast path, cache sharding) are under 50 lines each. The worker pool is the largest change, but even that is a refactor of one function, not an architectural rewrite. The codebase is small and well-factored enough that aggressive changes are still cheap. Do them now, before the Searchables layer is built on top.

On the Searchables Layer

The planned searchables table with value_json is fine for prototyping, but JSON is slow for filtering and useless for indexing. When you get there:

  • Use native columns for common scalar values (confidence, label)
  • Use FTS5 for text descriptions and tags
  • Use sqlite-vec or a dedicated vector column for embeddings
  • Consider a media_searchables junction table if you need many-to-many

Most importantly: run inference during scan, not on demand. When the scanner inserts a new media file, immediately enqueue it for classification. This amortizes cost and means search results are instant.

On Video Support

The process_file function uses image::ImageReader, which will fail on video files. When video support arrives, the scanner will need to probe files without decoding full frames. Consider using ffprobe-style metadata extraction (via ffmpeg-next or shelling out) rather than trying to decode videos through the image crate. The format column should be populated from the file extension as a fallback, which the code already does for HEIF.

8. Closing Thoughts

Akasha's backend is good MVP work with a few sharp edges that will cut deeply at scale. The developer has clearly thought about resumability, data integrity, and UI responsiveness — the hard problems are identified correctly. The remaining work is mostly mechanical optimization: batching, parallelism, and pagination. None of it requires throwing away the schema or the architecture.

The most important thing to do before building Searchables is to fix the scanner's throughput. If classification is layered on top of a scanner that takes an hour to walk 424k files, the user experience will be miserable regardless of how clever the AI models are. A fast, incremental, resumable scanner is the foundation everything else rests on.

Fix the batching and the mtime fast path first. They're low-effort, high-impact, and they set the tone for the rest of the backend.
